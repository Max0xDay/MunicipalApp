Overview - Data Structure Ecosystem

Purpose and Architecture

The Sidequest Municipal App implements a sophisticated data structure ecosystem designed to manage municipal service requests efficiently. The architecture separates concerns into node components (fundamental building blocks) and implementation structures (complete data structures).

Graph Structure - Relationship Network

Purpose

Manages network of municipal tickets with weighted relationships, revealing how service requests relate and influence each other.

Core Features

  • Undirected weighted graph
  • Automatic relationship building by category and location
  • Breadth-first and depth-first traversals
  • Minimum spanning tree computation (Prim's algorithm)
  • Related request tracking in ServiceRequest.RelatedRequestIDs

Key Operations

GraphNode AddOrUpdate(ServiceRequest) - O(1) node insertion
bool AddRelationship(string, string, double) - O(1) edge creation
void BuildRelationships(IEnumerable) - O(n²) batch relationship building
IReadOnlyList BreadthFirst(string, int depth) - O(V+E) traversal
IReadOnlyList DepthFirst(string) - O(V+E) traversal
IReadOnlyList GetRelatedRequests(string) - O(1) neighbor lookup
IReadOnlyList<GraphEdge> MinimumSpanningTree(string) - O(E log V) Prim's MST
void Clear() - O(1) graph reset

Relationship Building Algorithm

  1. Clear all existing RelatedRequestIDs
  2. Group requests by (Category, Location) pairs
  3. Within each group, connect all pairs with weight 1.0
  4. Update RelatedRequestIDs bidirectionally
  5. Trim excess capacity in lists

Traversal Implementations

  • BreadthFirst - Queue-based with depth limit, HashSet for visited tracking
  • DepthFirst - Stack-based, ordered by key descending for determinism
  • Both use StringComparer.OrdinalIgnoreCase for case-insensitive keys

Minimum Spanning Tree

  • Prim's algorithm with manual edge extraction
  • Maintains visited set and candidate edge list
  • ExtractMinimumEdge handles weight ties with lexicographic key comparison
  • Returns edges forming MST, excludes cycles

Application Context

ServiceRequestStatusForm pumps records from SQLite during refresh cycles, builds adjacency lists, then displays related requests, graph traversals, and MST visualizations. Helps citizens understand which incidents are linked and where efficient resolution pathways exist.

Heap Structure - Priority Management

Purpose

Implements min-heap for priority-based extraction, ensuring highest-priority tickets are always accessible in O(1) time.

Core Features

  • Array-based binary heap using List<HeapNode>
  • Higher priority values bubble up (max-heap behavior for urgency)
  • Timestamp-based tie-breaking favors earlier submissions
  • HeapifyUp and HeapifyDown maintain heap property
  • GetOrderedSnapshot for sorted views without destructive extraction

Key Operations

void Insert(ServiceRequest) - O(log n) insertion with bubble-up
ServiceRequest Peek() - O(1) view highest priority
ServiceRequest Extract() - O(log n) remove highest priority
IReadOnlyList GetOrderedSnapshot() - O(n log n) sorted copy
void Clear() - O(1) heap reset

Heap Property

For max-heap behavior (higher priority = more urgent):

  • Parent priority >= child priority
  • If priorities equal, earlier EnqueuedAt wins
  • Root always contains highest priority request

Heapify Algorithms

  • HeapifyUp - After insertion, bubble new node up until heap property restored
  • HeapifyDown - After extraction, trickle root replacement down to correct position
  • IsHigherPriority - Compares nodes: priority first, then EnqueuedAt for ties

Array Indexing

Parent of index i: (i - 1) / 2
Left child of i: i * 2 + 1
Right child of i: i * 2 + 2

Application Context

ServiceRequestStatusForm uses priority widget to spotlight top municipal fires. Heap recalculates after status changes, revealing urgent tickets without database re-queries. Ensures dispatch decisions stay agile during emergencies.

Queue Structure - Multi-Tier Priority

Purpose

Generic multi-level priority queue maintaining FIFO ordering within each priority tier, ensuring fair processing of same-priority items.

Core Features

  • Generic type parameter: PriorityCategoryQueue<T> where T : class
  • SortedDictionary<int, Queue<T>> for priority lanes
  • Lower integer = higher priority (dequeues first)
  • FIFO within each priority level
  • Automatic lane creation and cleanup

Key Operations

void Enqueue(T item, int priority) - O(log k) where k = active priorities
T Dequeue() - O(log k) remove highest priority item
T Peek() - O(log k) view highest priority item
bool Contains(int priority) - O(log k) priority existence check
int GetPriorityCount(int) - O(log k) items in priority tier
IEnumerable<int> GetActivePriorities() - O(k) all active priority levels
void Clear() - O(1) queue reset
int Count - O(1) total item count

Data Structure Design

  • priorityQueues - SortedDictionary maps priority to Queue<T>
  • priorityCounts - Dictionary tracks count per priority for fast lookup
  • Automatic lane removal when empty
  • SortedDictionary.Keys.First() gives highest priority in O(log k)

Fairness Guarantee

Unlike a single heap where ties are arbitrary, this structure ensures items with equal priority are processed in submission order. Critical for municipal support to avoid blindsiding frontline staff.

Application Context

Backs ServiceRequestHeap when easy priority grouping with FIFO semantics is needed. ServiceRequestStatusForm reads snapshots for priority sidebar, communicating both urgency and fairness without database overhead.

Suggestion Retrieval Algorithm

  • Step 1: Traverse trie letter-by-letter to prefix endpoint
  • Step 2: Recursively collect all terminal suggestions from subtree
  • Step 3: Sort by frequency (descending), then by length (ascending)
  • Step 4: Return top N results (default: 5)
  • Empty prefix: Returns empty list (requires user input)
  • No match: Returns empty list (prefix not in trie)

Dynamic Learning Mechanism

When users submit custom issue descriptions (5+ characters), the system automatically inserts them into the trie with frequency 1. Repeated submissions increment the frequency counter, causing popular phrases to bubble up in future suggestions. This creates a self-improving prediction system tailored to the specific municipality's most common issues.

Application Context

The ReportIssues form integrates this trie for real-time autocomplete as users type issue descriptions. After typing just two letters like "Bu", the system instantly suggests "Burst Pipe" and "Burst Water Main" ranked by how often residents have submitted those phrases. This standardizes issue descriptions across reports, streamlining ticket categorization and dispatch routing for municipal staff.

AVL Node O(n) O(log n) stack Height cached per node
BST Node O(n) O(h) stack No balancing overhead
Binary Tree Node O(n) O(w) queue Level tracking per node
Red-Black Node O(n) O(log n) stack Parent pointer + color bit
Tree Node O(n) O(h) recursion Variable child list
Graph Node O(V+E) O(V) visited set Adjacency dictionary per node
Heap Node O(n) O(1) Array-based heap
Priority Queue O(n) O(k) k = number of priority tiers
Trie Structure O(n * m) O(p) recursion Dictionary per node + frequency map

Best Use Cases

  • AVL - Frequent searches, balanced guarantees
  • BST - Simple implementation, random insertions
  • Red-Black - Category clustering with balance
  • Heap - Priority-based extraction
  • Graph - Relationship discovery
  • Trie - Prefix-based autocomplete, predictive text

Worst Case Scenarios

  • BST - Sorted input degenerates to O(n) list
  • Binary Tree - No ordering, O(n) search always
  • Graph - Dense graphs increase edge traversal
  • AVL - ID search requires full traversal
  • Trie - Memory overhead with many unique strings

Balancing Mechanisms

  • AVL - Rotations after insert (4 cases)
  • Red-Black - Recoloring + rotations (6 cases)
  • Heap - Heapify up/down after modifications
  • BST - No balancing
  • Trie - No balancing needed
